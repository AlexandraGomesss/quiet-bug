# User Flows

## Purpose

This document describes how users move through Quiet Bug.

User flows help us understand what the user does step by step before we start coding the app.

## Main Practice Flow

This is the main flow for a junior developer practising a live coding exercise.

1. User opens Quiet Bug.
2. User reads the homepage message.
3. User clicks "Start Practicing".
4. User chooses "Java Practice".
5. User sees a list of junior Java exercises.
6. User selects one exercise.
7. User reads the problem statement.
8. User fills the thinking checklist.
9. User writes code in the code area.
10. If the user gets stuck, they click "I'm stuck".
11. The app shows one hint at a time.
12. User continues working on the solution.
13. User opens the oral explanation section.
14. User practises explaining the approach out loud.
15. User compares their answer with the suggested solution.
16. User reads the interview feedback.
17. User chooses another exercise or returns to the practice page.

## Panic Recovery Flow

This flow is used when the user freezes or does not know how to start.

1. User is on an exercise page.
2. User feels blocked.
3. User clicks "I'm stuck".
4. The app does not show the full solution immediately.
5. The app shows a small recovery question or hint.
6. User thinks about the hint.
7. User can click again to reveal the next hint.
8. The app guides the user back to the problem structure.

## Panic Recovery Questions

Examples of recovery questions:

- What is the input?
- What is the expected output?
- Do you need to repeat an action?
- Do you need a loop?
- Do you need a helper variable?
- Can you explain the logic in plain English first?
- What is the simplest version of the problem?
- What edge case should you consider?

## Speak While Coding Flow

This flow helps the user practise communication during a technical interview.

1. User opens an exercise.
2. User reads the problem.
3. User thinks about the solution.
4. User opens the "Speak While Coding" section.
5. The app shows an example explanation.
6. User practises saying the explanation out loud.
7. User adapts the explanation to their own words.

## Soft Skills Flow

This flow helps the user practise non-coding interview moments.

1. User opens the Soft Skills page.
2. User chooses a topic.
3. User reads a common junior interview question.
4. User sees guidance about what the employer is checking.
5. User reads a weak answer example.
6. User reads a stronger answer example.
7. User writes or practises their own answer.

## Employer Mindset Flow

This flow helps the user understand what companies usually expect from junior developers.

1. User opens the Employer Mindset section.
2. User reads a topic, such as "What employers expect from juniors".
3. User learns what matters in real teams.
4. User connects those expectations with interview behaviour.

## First MVP Flow

For MVP v0, the most important flow is:

1. Open homepage.
2. Go to Java Practice.
3. Select exercise.
4. Read prompt.
5. Fill thinking checklist.
6. Write code.
7. Use "I'm stuck" if needed.
8. Read oral explanation.
9. Compare with solution.
10. Read feedback.

## Notes

The MVP should keep the flow simple.

The user should never feel lost inside the app.

Every exercise page should answer these questions clearly:

- What do I need to solve?
- How do I start thinking?
- What do I do if I freeze?
- How do I explain this in an interview?
- What can I learn from the solution?
